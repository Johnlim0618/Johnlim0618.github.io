Atomic Red Team Validation Workflow

I used Atomic Red Team to better understand how ATT&CK techniques generate telemetry, how that telemetry appears in Windows logs and SIEM searches, and whether the detections I expected to work were actually visible in the lab.

Overview

Instead of only reading about adversary techniques, I wanted to validate behavior in a controlled environment. This project helped me build a repeatable process for running safe tests, checking log visibility, and documenting whether a detection or alert behaved as expected.

Objectives

  • Test adversary-like activity safely in a lab
  • Verify whether expected telemetry appears in logs
  • Compare expected detections against actual SIEM visibility

Tools Used

  • Atomic Red Team
  • Windows logs
  • Splunk
  • Wazuh
  • MITRE ATT&CK

Environment

I used a lab environment where Windows activity could be observed through log sources and detection platforms. The emphasis was not just on “running a test,” but on validating whether the underlying telemetry was present and useful.

What I Did

  • Selected techniques that could be safely tested in a lab setting
  • Ran test activity and documented expected telemetry
  • Checked logs and SIEM searches for the related events
  • Compared what I expected to see versus what was actually visible

Key Findings

  • Detection coverage is only as good as the underlying visibility
  • Not every expected event appears the way you assume it will
  • Validation work is critical before trusting a detection in practice

Example Artifact

Validation Checklist
1. Run the Atomic Red Team test safely in the lab
2. Confirm expected Windows event logs were generated
3. Check if Sysmon / WEF forwarded the activity
4. Search Splunk / Wazuh for the related telemetry
5. Confirm whether an alert triggered
6. Document gaps, false negatives, and tuning ideas

What I Learned

This project made me much more careful about assuming a detection “works” just because the rule exists. Real validation requires checking whether the logs, forwarding, and alert logic actually line up.
