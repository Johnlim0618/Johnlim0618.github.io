False Positive Tuning Exercise

This project focused on understanding how suspicious-looking activity can still turn out to be benign, and why tuning detections is necessary to improve signal quality without removing useful coverage.

Overview

In a SOC, not every alert is a real incident. This exercise helped me think through how an analyst should respond to a detection that looks serious at first, gather the right context, and decide whether the alert should be escalated, closed as benign, or tuned for the future.

Objectives

  • Understand how false positives happen in real detection workflows
  • Practice structured triage and documentation
  • Learn how tuning can improve analyst efficiency without losing useful signal

Tools Used

  • Splunk
  • Windows process and event context
  • Basic triage documentation workflow
  • Detection engineering concepts

Environment

This exercise was based on reviewing suspicious process behavior in a lab environment and thinking through how a detection might fire on activity that appears malicious at first glance. The goal was not just to close the alert, but to understand why it fired and how future noise could be reduced.

What I Did

  • Reviewed an alert path that resembled suspicious LSASS-related access behavior
  • Looked at process context, parent-child relationships, and trusted binary indicators
  • Compared the activity against expected host behavior and surrounding telemetry
  • Documented why the alert appeared suspicious and what evidence supported a benign conclusion

Key Findings

  • Single-event detections can be noisy when environment context is missing
  • Benign system behavior can look malicious if a rule is too broad
  • Tuning should be evidence-based and as narrow as possible
  • Analysts need to document why they trust a tuning decision

Example Artifact

Alert: Suspicious LSASS Access
Initial reason: Process attempted access pattern associated with credential dumping
Reviewed:
- Parent process
- Signed binary status
- Host baseline behavior
- User context
Conclusion:
Benign system activity from trusted process path
Tuning idea:
Exclude signed trusted process under validated parent/host conditions

What I Learned

This project taught me that a good analyst should not just ask “is this malicious?” but also “why did this alert fire?” and “how can we improve this detection without weakening it?” That mindset is important for reducing alert fatigue and making investigations more efficient.
