Splunk + Wazuh Detection Lab

I built this lab to get hands-on practice with security monitoring, detection validation, and log-based investigation in a small SOC-style environment. The goal was to centralize Windows telemetry, review alerts, and better understand how suspicious activity appears across SIEM and endpoint-focused tooling.

Overview

This project focused on building a practical home lab using Splunk and Wazuh with Windows-based telemetry. I wanted a space where I could test detections, analyze event logs, and document the investigation process the way an entry-level analyst would in a real environment.

Objectives

  • Centralize Windows security-related logs for investigation practice
  • Understand how SIEM visibility supports triage and incident analysis
  • Validate that suspicious activity can be observed and investigated properly

Tools Used

  • Splunk
  • Wazuh
  • Windows Event Logs
  • Sysmon / WEF concepts
  • PowerShell

Environment

The lab was built around Windows systems and centralized visibility tooling. I used the environment to monitor authentication events, PowerShell-related activity, and privilege-relevant telemetry while comparing raw logs against what actually surfaced inside the SIEM workflow.

What I Did

  • Configured a lab environment to collect and review Windows telemetry
  • Used Splunk searches to identify patterns like repeated failed logons followed by success
  • Reviewed Wazuh alerts to understand how host-based detections and event visibility overlap
  • Documented how logs moved from raw telemetry into usable detection and triage workflows

Key Findings

  • Good visibility depends on both log collection and log quality
  • Authentication activity is much easier to investigate when events are centralized and searchable
  • Signal quality improves when detections are paired with context, not just single events

Example Artifact

index=wineventlog (EventCode=4625 OR EventCode=4624)
| eval user=coalesce(TargetUserName, Account_Name)
| stats count(eval(EventCode=4625)) as failed_count
        count(eval(EventCode=4624)) as success_count
        min(_time) as first_seen
        max(_time) as last_seen
        by user host
| where failed_count >= 5 AND success_count >= 1
| convert ctime(first_seen) ctime(last_seen)

What I Learned

This project helped me understand the difference between simply collecting logs and actually using them for triage. It also showed me how detection logic, event visibility, and analyst context all work together.
