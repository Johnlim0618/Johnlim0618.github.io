Windows Event Investigation Practice

This project focused on building stronger investigation habits around common Windows security events. I used this work to improve my understanding of how authentication activity, PowerShell execution, and privileged group changes appear in logs and how an analyst would begin triaging them.

Overview

Windows event logs are one of the most important sources of visibility in security operations. I used this project to study high-value events like failed logons, account changes, and PowerShell activity so I could better understand what to look for, what context matters, and how one event can lead into a broader investigation.

Objectives

  • Improve familiarity with common Windows security event IDs
  • Practice investigation steps for suspicious authentication and account-related activity
  • Understand how context changes the meaning of an event

Tools Used

  • Windows Event Viewer
  • Windows Security Event Logs
  • PowerShell
  • Splunk
  • MITRE ATT&CK concepts

Environment

I used a Windows-based lab environment where security events could be generated, reviewed, and searched. The goal was to build comfort around interpreting Windows telemetry in a way that supports investigation, detection validation, and escalation decisions.

What I Did

  • Reviewed Event ID 4625 for failed logon activity and looked for repeated authentication failures
  • Studied Event ID 4732 to understand how privileged group membership changes appear in the Security log
  • Reviewed PowerShell-related logging to better understand script execution visibility and analyst context
  • Used those events to practice basic triage steps and think through how each event might fit into a larger attack chain

Key Findings

  • A single Windows event is rarely enough on its own and needs supporting context
  • Authentication events become much more meaningful when tied to time, host, and user behavior
  • Privileged group changes are especially important because they can signal persistence or privilege escalation
  • PowerShell logging can provide strong visibility, but only if logging is configured and collected properly

Example Artifact

$ids = 4624,4625,4732
Get-WinEvent -LogName Security |
Where-Object { $ids -contains $_.Id } |
Select-Object TimeCreated, Id, ProviderName, Message |
Format-Table -AutoSize

What I Learned

This project helped me get more comfortable moving from “I know this event ID exists” to “I know why this matters, what context I need, and what I would investigate next.” It also reinforced that event triage is as much about narrowing possibilities as it is about identifying obvious threats.
